Flask 源码阅读: 请求流程

Posted on 2018-09-09 | In Flask |

阅读 Flask 源码的系列笔记, 这一篇说的是 Flask 的请求流程

这里是用 Debug 模式运行所以出现的会是 DebugApplication 对于其他情况都会调用不同的 Application. 例如, 多线程是 ThreadedWSGIServer, fork 是 ForkingWSGIServer等, 但是除了具体等 多线程 和 fork 操作不一样之外, 其他等流程大致也是一样的. 就不详细写了.

以后有时间在看看要不要上图之类的.

Flask 请求流程

启动代码

正文

  1. 首先线程被唤醒, 调用 Thread.run 方法, 然后由 ThreadedWSGIServer 处理该线程的请求.

    ThreadedWSGIServer 是一个执行线程的 WSGI 服务器, 在启动服务的时候如果选择了多线程就会初始化并返回这个的实例

  2. 调用 ThreadedWSGIServer 的父类 ThreadingMixIn 的 process_request_thread 方法.

    这是一个用于处理新线程中的每个请求的混合类, 而 process_request_thread 和 BaseServer 一样, 但是不同的是它是请求线程里的, 而且做了异常处理.

  3. process_request_thread 首先会调用 BaseServer.finish_request 方法, 使用 self.RequestHandlerClass (即 BaseRequestHandler)开始实例化一个 request

    ThreadedWSGIServer 继承了 ThreadingMixIn 和 BaseWSGIServer, 而 BaseWSGIServer -继承-> HTTPServer -继承-> TCPServer -继承-> BaseServer.
    ThreadedWSGIServer 是 BaseServer 的子类, 所以能调用 BaseServer 的方法,它也是服务器类的超级父类.

  4. 在实例化 BaseRequestHandler 的时候, 会执行 BaseRequestHandler.handle 完成后执行 BaseRequestHandler.finish
  5. 因为在初始化包werkzeug.serving 的时候, 会将 WSGIRequestHandler 类赋值给BaseRequestHandler , 所以现在会执行 WSGIRequestHandler 的 handle 方法.
  6. WSGIRequestHandler.handle 会调用父类 BaseHTTPRequestHandler 的 handle 方法. BaseHTTPRequestHandler.handle 会调用 WSGIRequestHandler.handle_one_request 方法处理单个的 http 请求, 如果有多个请求也会由这里处理. 使用 while 循环执行 handle_one_request 直到连接关闭.
  7. 在 WSGIRequestHandler.handle_one_request 中会获取原始请求行, 然后调用BaseHTTPRequestHandler.parse_request 去解析一个 http 请求的请求行, 请求头, 请求路径等. 然后再调用 WSGIRequestHandler.run_wsgi
  8. 在 WSGIRequestHandler.run_wsgi 中会调用内部方法 execute , 在 execute 中就会实例化 werkzeug 的 Application. 首先会初始化 BaseRequest 和 将 DebuggedApplication.debug_application 方法分配给 response ,给 request 并根据 request 的参数调整 response, 再返回 response 方法结果的迭代器.

    在实例化的时候会初始化 BaseRequest, BaseRequest 是非常基本的请求对象。它不会实现实体标记解析或缓存控制等高级内容。除非它是使用populate_request设置为False创建的, 不然它会使用WSGI环境作为第一个参数创建请求对象，并将自己添加到WSGI环境中作为 werkzeug.request.
    DebuggedApplication.debug_application 会运行 Application 并保存跟踪信息

  9. Application 会调用 Flask 实例, 在 call 方法中, WSGI 服务器会调用 Flask Application 对象到 WSGI Application.在 call 方法中会调用的 wsgi_app, 它可以包装到应用中间件。
  10. 在 wsgi_app 刚开始就会通过 Flask.request_context 构建 请求上下文, Flask.request_context 会初始化 RequestContext , 然后将 请求上下文 推入应用上下文, 然后调用 Flask.full_dispatch_request.

    在初始化的时候 RequestContext 会提供一个 match_request 方法, 这个方法可以被子类重写以挂钩到请求的匹配。

  11. Flask.full_dispatch_request 会调度请求，最重要的是执行请求前后处理以及HTTP异常捕获和错误处理。
    1. 首先调用 try_trigger_before_first_request_functions 方法, 这个方法会调用装饰了 before_first_request 这个装饰器的方法.并确保只调用一次.
    2. 调用向 Application 和当前蓝图注册的 url_value_preprocessors。然后调用在 Application 和蓝图中注册的before_request_funcs。 如果任何before_request() 处理程序返回非None值，则处理该值就好像它是视图中的返回值一样，并且停止进一步的请求处理。
    3. 调用 dispatch_request 进行请求调度, 执行请求的方法.
  12. 执行请求方法
  13. 请求方法返回结果给 dispatch_request, dispatch_request 返回结果给 full_dispatch_request.
  14. full_dispatch_request 继续执行 finalize_request 方法. 它会通过将请求转换为响应并调用后处理函数来最终确定响应。 finalize_request 调用 make_response 将结果转换为 response. 然后调用 process_response 方法执行装饰了 after_request 的方法, 在响应发送到 WSGI 服务器之前修改响应.
  15. 然后将结果返回到 wsgi_app, 它会继续返回, 并将请求上下文弹出应用上下文.
  16. 结果返回到 DebuggedApplication.debug_application 它会返回一个迭代器迭代返回的结果. 然后 execute 方法会调用 write 方法将结果发送回客户端

    write 方法会先后返回响应行, 响应头, 响应体

  17. 在响应回客户端后会一直返回并且 close, 等待下一个请求.
